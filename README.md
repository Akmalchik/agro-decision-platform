# Agro Decision Platform

Фундамент государственной веб-платформы поддержки принятия решений в сельском хозяйстве Республики Узбекистан.

## Архитектура

- `src/app` — маршруты, страницы и HTTP API (Next.js App Router)
- `src/components` — общие UI-компоненты и providers
- `src/services` — прикладные сценарии и бизнес-логика
- `src/database` — Prisma client и репозитории
- `src/modules` — доменные сущности и контракты, не зависящие от UI и инфраструктуры
- `src/gis` — изолированные GIS-контракты, PostGIS-доступ и компоненты карты
- `src/api` — схемы входных данных и API-контракты
- `src/types` — общие TypeScript-типы
- `src/utils` — чистые вспомогательные функции
- `prisma` — модель данных и миграции

Зависимости направлены от страниц к сервисам, а сервисы зависят от доменных контрактов, не от Prisma. Реализации репозиториев находятся в инфраструктурных слоях `database` и `gis`.

Геометрия хранится нативно как PostGIS `geometry(Geometry, 4326)`. Пространственные фильтры, пересечения, расстояния, преобразования систем координат и агрегации должны выполняться функциями PostGIS в GIS-репозиториях. JavaScript получает только итоговые данные, необходимые вызывающему коду.

Текущий GIS-сценарий использует PostGIS-реализацию `PlotGeoJsonSource`. Она формирует GeoJSON через `ST_AsGeoJSON`, а границы обзора — через `ST_Extent`, `ST_XMin`, `ST_YMin`, `ST_XMax` и `ST_YMax`. Компонент карты не выполняет пространственных вычислений.

Почвенные характеристики отделены от участка необязательным отношением `Plot` → `SoilProfile`. Это позволяет расширять почвенный модуль независимо, не меняя основной контракт участка.

## Локальный запуск

Требуется установленный и запущенный Docker Desktop либо совместимый Docker runtime.

1. Установить зависимости: `npm install`.
2. Скопировать `.env.example` в `.env`. Для стандартного Compose задайте `DATABASE_URL` и `DIRECT_URL` одинаковым значением: `postgresql://agro:agro@localhost:5432/agro?schema=public`.
3. Запустить PostgreSQL/PostGIS: `npm run db:start`.
4. Создать Prisma Client: `npm run db:generate`.
5. Применить миграции: `npm run db:migrate`.
6. Идемпотентно записать тестовый участок: `npm run db:seed`.
7. Запустить приложение: `npm run dev`.

Повторный `npm run db:seed` обновляет тот же UUID и не создаёт дубликаты. Команда `npm run db:clear-test` удаляет только тестовый участок с известным UUID; связанный почвенный профиль удаляется каскадно. Остановить базу можно командой `npm run db:stop`.

## Маршруты

- `/` — главная
- `/map` — карта с тестовым земельным участком
- `/plot/[id]` — карточка участка через сервисный слой
- `/api/health` — проверка доступности приложения

Production-инструкция для Vercel + Supabase находится в [`docs/deployment.md`](docs/deployment.md).
